The **Model Context Protocol (MCP)** is an open standard that standardizes how AI applications connect to external tools, data sources, and workflows. Often described as "USB-C for AI," it provides a common language for AI assistants to discover and invoke capabilities across different services [citation:1].

## Core Architecture

An MCP setup consists of three core components:

- **Host**: The LLM application (like Claude Desktop, VS Code, or SkillDeck) that runs the model and coordinates MCP connections.
- **MCP Client**: A connector that the host launches to maintain a 1:1 session with a server.
- **MCP Server**: A service endpoint that exposes capabilities, such as creating GitHub issues or fetching database records [citation:1].

Through this architecture, AI systems can access live data and trigger operations in real time via standardized JSON-RPC 2.0 messages defined by the MCP specification [citation:8].

## Why MCP Matters

MCP has moved from niche adoption to widespread use, with major platforms like OpenAI, Microsoft, and Google now supporting it natively [citation:1]. Its rapid adoption stems from several key benefits:

### 1. **Interoperability and Standardization**
Before MCP, every LLM provider and tool creator had to build custom integrations. MCP provides a standardized method for capability discovery and JSON-RPC messaging, eliminating countless one‑off integrations [citation:3].

### 2. **Enterprise-Ready Controls**
MCP supports transport‑level authorization so clients can call restricted servers on a user's behalf. While the protocol doesn't mandate role‑based access control, hosts and servers can implement these features, along with consent prompts and per‑tool identity scoping [citation:1].

### 3. **Dynamic Tool Discovery**
Unlike static function‑calling approaches, MCP servers can advertise their capabilities at runtime, allowing clients to request a list of available tools and receive a structured response [citation:4].

## How MCP Servers Work

Each MCP server exposes one or more **tools** – discrete capabilities that the AI can invoke. For example, a filesystem server might expose `read_file`, `write_file`, and `list_directory` tools [citation:5].

When an AI agent needs to perform an operation:

1. The host sends a tool request to the MCP client
2. The client forwards it to the appropriate server via the configured transport (stdio for local processes, SSE for remote servers)
3. The server executes the operation and returns a structured response
4. The client passes the result back to the AI model

This entire exchange happens with **sub‑50ms latency** in optimized implementations, making real‑time AI assistance possible [citation:8].

## Security Considerations

MCP's rapid adoption has surfaced important security questions. The original specification lacked authentication mechanisms, but modern implementations address this through:

- **OAuth integration** for user‑centric authorization
- **Read‑only modes** for sensitive operations
- **Tool‑level permissions** that limit what each server can do
- **Audit logging** of all tool calls for compliance [citation:3][citation:8]

However, challenges remain – particularly around **prompt injection**, where malicious data returned by a server could override the AI's intended behavior. This remains an active area of development [citation:6].

## Why MCP Matters for SkillDeck

SkillDeck embraces MCP as the standard for extending agent capabilities. By supporting MCP, you can:

- **Add any capability** via a server – from filesystem access to GitHub integration
- **Use the same server** with other MCP‑compatible clients
- **Control exactly what resources** are exposed, since servers run locally

The ecosystem now includes servers for everything from web scraping and database access to CRM integration and crypto market data [citation:1][citation:7].
