## Issue 1 ŌĆö Smart Auto-Name via Local Ollama Agent

### Current behaviour
`use-agent-stream.ts ŌåÆ autoNameConversation()` grabs the **first user message** and truncates it to 60 characters. There is no LLM involved, no re-triggering, and no goal extraction.

```ts
// Current (na├»ve)
const raw = firstUserMsg.content.trim().slice(0, 60)
const title = raw.charAt(0).toUpperCase() + raw.slice(1)
```

### Root cause
The `autoNameAttempted` ref is a `Set<string>` keyed by `conversationId`. Once an attempt is recorded it never fires again, regardless of how many messages have been sent or whether the topic changed.

### Implementation

#### 1.1 ŌĆö Track message count for re-trigger

**File:** `src/hooks/use-agent-stream.ts`

Replace the bare `autoNameAttempted` Set with a map that stores the **last message count at which a naming attempt was made**:

```ts
// Before
const autoNameAttempted = useRef<Set<string>>(new Set())

// After
const autoNameState = useRef<Map<string, { lastCount: number }>>(new Map())
```

In the `'persisted'` event handler, after invalidating queries, fetch the current message count and decide whether to run:

```ts
case 'persisted':
  queryClient.invalidateQueries({ queryKey: ['messages', conversationId], exact: false })
  queryClient.invalidateQueries({ queryKey: ['conversations'], exact: false })

  const messages = queryClient.getQueryData<MessageData[]>(['messages', conversationId])
  const msgCount = messages?.length ?? 0
  const state = autoNameState.current.get(conversationId)
  const lastCount = state?.lastCount ?? 0

  // Fire on first message OR every 5 new messages
  const shouldRename = lastCount === 0 || (msgCount - lastCount) >= 5
  if (shouldRename) {
    autoNameState.current.set(conversationId, { lastCount: msgCount })
    autoNameConversation(messages ?? [])
  }
  break
```

#### 1.2 ŌĆö Call a local Ollama agent for goal extraction

**File:** `src/hooks/use-agent-stream.ts` ŌĆö replace `autoNameConversation` body.

```ts
const autoNameConversation = async (messages: MessageData[]) => {
  try {
    const convoQueries = queryClient.getQueriesData<Array<{ id: string; title: string | null }>>({ queryKey: ['conversations'] })
    const conversations = convoQueries.flatMap(([, data]) => data ?? [])
    const currentConvo = conversations.find((c) => c.id === conversationId)
    if (!currentConvo) return

    // Collect first 3 user messages as context
    const userMessages = messages
      .filter((m) => m.role === 'user')
      .slice(0, 3)
      .map((m) => m.content.trim())
      .join('\n---\n')

    if (!userMessages) return

    // Call Ollama locally ŌĆö use the lightest available model
    const ollamaRes = await commands.listOllamaModels()
    const ollamaModel = ollamaRes.status === 'ok' && ollamaRes.data.length > 0
      ? ollamaRes.data[0].id   // cheapest/first available
      : null

    let title: string

    if (ollamaModel) {
      // Ask Ollama to produce a short goal-oriented title
      const prompt = [
        'You are a conversation title generator.',
        'Given the opening messages below, output ONLY a short title (3ŌĆō7 words, no punctuation) that captures the user\'s goal.',
        'Do NOT explain. Do NOT use quotes. Output the title only.',
        '',
        userMessages
      ].join('\n')

      const res = await fetch('http://localhost:11434/api/generate', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ model: ollamaModel, prompt, stream: false })
      })

      if (res.ok) {
        const data = await res.json()
        title = (data.response as string).trim().slice(0, 60)
      } else {
        // Fallback: simple truncation
        title = userMessages.split('\n')[0].slice(0, 60)
      }
    } else {
      title = userMessages.split('\n')[0].slice(0, 60)
    }

    if (!title) return
    title = title.charAt(0).toUpperCase() + title.slice(1)

    const renameRes = await commands.renameConversation(conversationId, title)
    if (renameRes.status === 'ok') {
      queryClient.invalidateQueries({ queryKey: ['conversations'], exact: false })
    }
  } catch (error) {
    console.error('Failed to auto-name conversation:', error)
  }
}
```

> **Note:** The fetch to `localhost:11434` is safe inside a Tauri desktop app. Add `http://localhost:11434` to `tauri.conf.json ŌåÆ allowlist.http.scope` if CSP blocks it.

#### 1.3 ŌĆö (Optional) Expose a Tauri command for the naming call

If you want to avoid browser-side `fetch` to Ollama, add a Rust command `smart_name_conversation(conversation_id, messages)` in `src-tauri` that uses the existing `OllamaProvider` and fires a non-streaming single-turn completion. This keeps network calls inside Rust and respects any existing Ollama port configuration.

## Issue 2 ŌĆö Model Selector Does Not Update Active Conversation Profile

### Current behaviour
`ModelSelector` in `right-panel.tsx` has its own local `useState(currentModelId)`. Changing the select calls `setSelected` ŌĆö that's it. The profile in the database is **never updated**.

```tsx
// Current ŌĆö dead-end local state
const [selected, setSelected] = useState(currentModelId)
// ...
<Select value={safeSelected} onValueChange={setSelected}>
```

### Root cause
`onValueChange` only updates `selected` in component memory. It never calls `commands.updateProfile(...)` or `useUpdateProfile()`.

### Implementation

**File:** `src/components/layout/right-panel.tsx`

1. Import `useUpdateProfile` from `use-profiles`.
2. Change `ModelSelector` to accept `profileId` as a prop.
3. On value change, call `updateProfile` mutation and show a toast.

```tsx
// Props extended
function ModelSelector({
  profileId,
  provider,
  currentModelId
}: {
  profileId: string
  provider: string
  currentModelId: string
}) {
  const { data: models = [], isLoading } = useAvailableModels(provider)
  const [selected, setSelected] = useState(currentModelId)
  const updateProfile = useUpdateProfile()

  const displayModels = models.length > 0 ? models : [currentModelId]
  const safeSelected = displayModels.includes(selected) ? selected : displayModels[0]

  const handleChange = async (modelId: string) => {
    setSelected(modelId)
    updateProfile.mutate(
      { id: profileId, model_id: modelId },
      {
        onSuccess: () => toast.success(`Model switched to ${modelId}`),
        onError: (e) => {
          setSelected(currentModelId) // rollback
          toast.error(`Failed to switch model: ${e}`)
        }
      }
    )
  }

  return (
    <div className="space-y-1">
      <label htmlFor="model-select" className="text-xs text-muted-foreground">
        Model {isLoading && <span className="opacity-50">(loadingŌĆ”)</span>}
        {updateProfile.isPending && <span className="opacity-50 ml-1">(savingŌĆ”)</span>}
      </label>
      <Select value={safeSelected} onValueChange={handleChange} disabled={updateProfile.isPending}>
        <SelectTrigger id="model-select" className="h-7 text-xs">
          <SelectValue placeholder="Select model" />
        </SelectTrigger>
        <SelectContent>
          {displayModels.map((m) => (
            <SelectItem key={m} value={m} className="text-xs">{m}</SelectItem>
          ))}
        </SelectContent>
      </Select>
    </div>
  )
}
```

4. In `SessionTab`, pass `profile.id` to `ModelSelector`:

```tsx
<ModelSelector
  profileId={profile.id}        // ŌåÉ new
  provider={effectiveProvider}
  currentModelId={profile.model_id}
/>
```

5. Sync local state with incoming prop (handles external changes):

```tsx
useEffect(() => {
  setSelected(currentModelId)
}, [currentModelId])
```

## Issue 4 ŌĆö Structured Output Fallback When Model Doesn't Support Tools

### Current behaviour
`AgentLoop.build_request()` unconditionally passes `tools` to the provider. If the model responds with a 400 (e.g., `gemma3:12b does not support tools`), the error bubbles up as an unrecoverable agent failure.

### Implementation

#### 5.1 ŌĆö Detect tool-unsupported error and retry with structured output

**File:** `src-tauri/skilldeck-core/src/agent/loop.rs`

In the `loop`, wrap the `provider.complete(request)` call to detect the "does not support tools" error and re-issue with a structured output prompt:

```rust
let stream = match tokio::select! {
    res = self.provider.complete(request.clone()) => res,
    _ = self.cancel_token.cancelled() => { /* ... */ }
} {
    Ok(s) => s,
    Err(CoreError::ProviderError { message, .. })
        if message.contains("does not support tools")
            || message.contains("tool_use")
            || message.contains("function calling") =>
    {
        // Retry with structured output fallback
        let fallback_request = self.build_structured_output_request(&request)?;
        tokio::select! {
            res = self.provider.complete(fallback_request) => res?,
            _ = self.cancel_token.cancelled() => { /* ... */ }
        }
    }
    Err(e) => return Err(e),
};
```

#### 5.2 ŌĆö Implement `build_structured_output_request`

```rust
fn build_structured_output_request(
    &self,
    original: &CompletionRequest,
) -> Result<CompletionRequest, CoreError> {
    // Serialise tools as a JSON schema block injected into the system prompt
    let tools_json = serde_json::to_string_pretty(&self.tools)
        .map_err(|e| CoreError::Internal { message: e.to_string() })?;

    let tool_instructions = format!(
        r#"
You do not have native tool-calling. Instead, when you need to call a tool, output ONLY a JSON object with this exact schema and nothing else:

{{"tool_call": {{"name": "<tool_name>", "arguments": {{...}}}}}}

Available tools:
```json
{tools_json}
```

After a tool result is provided, continue responding normally.
When you do not need a tool, respond normally without any JSON wrapper.
"#,
        tools_json = tools_json
    );

    let augmented_system = match &original.system {
        Some(s) => format!("{}\n\n{}", s, tool_instructions),
        None => tool_instructions,
    };

    Ok(CompletionRequest {
        system: Some(augmented_system),
        tools: vec![], // strip native tools
        tools_toon: None,
        ..original.clone()
    })
}
```

#### 5.3 ŌĆö Parse structured output tool calls in process_stream

**File:** `src-tauri/skilldeck-core/src/agent/loop.rs` ŌĆö in `process_stream`, after accumulating `content`:

```rust
// After stream is exhausted, check if entire content is a structured tool call
if tool_calls.is_empty() {
    if let Some(tc) = Self::try_parse_structured_tool_call(&content) {
        tool_calls.push(tc);
        content.clear(); // the "content" was the tool call JSON, not a real message
    }
}
```

```rust
fn try_parse_structured_tool_call(content: &str) -> Option<ToolCall> {
    let trimmed = content.trim();
    // Accept fenced JSON blocks or bare JSON
    let json_str = if trimmed.starts_with("```") {
        trimmed
            .trim_start_matches("```json")
            .trim_start_matches("```")
            .trim_end_matches("```")
            .trim()
    } else {
        trimmed
    };

    let val: serde_json::Value = serde_json::from_str(json_str).ok()?;
    let tc = val.get("tool_call")?;
    let name = tc.get("name")?.as_str()?.to_string();
    let arguments = tc.get("arguments")?.clone();

    Some(ToolCall {
        id: uuid::Uuid::new_v4().to_string(),
        function: crate::traits::FunctionCall {
            name,
            arguments: serde_json::to_string(&arguments).unwrap_or_default(),
        },
    })
}
```

#### 5.4 ŌĆö Flag the fallback in logs and optionally in the conversation metadata

Add a `tracing::warn!` when the fallback is triggered so it's visible in logs:

```rust
tracing::warn!(
    model = %self.model_id,
    "Model does not support native tools ŌĆö falling back to structured output"

```

Optionally surface this to the UI by emitting an agent event type `ToolFallback` so the session panel can show a small notice.
