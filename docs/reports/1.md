### Issue 0 — Tool call card: non-copyable, icon-synthesized, expandable

**File:** `src/components/conversation/tool-call-card.tsx`

**Root cause:** The card uses `<pre>` tags with raw JSON, making content trivially selectable. There's no synthesized description or icon differentiation per tool type; content is always directly readable.

**Changes:**

1. **Disable text selection** on the entire card container and its collapsed header:
   ```tsx

   className={cn('... select-none', ...)}
   ```

2. **Add a synthesized readable description** computed from the tool name and arguments. Create a `synthesizeDescription(name, args)` helper that maps common tool names to human-readable phrases:

   - `write_file` + `{path}` → `"Write to: path/to/file.txt"`
   - `run_shell` + `{command}` → `"Run: <command>"`
   - `http_request` + `{url}` → `"HTTP GET/POST: url"`

   Add this description as muted secondary text after the tool name in the header button.

### Issue 3 — Shiki code block: oversized horizontal scrollbar + wrong background color on scroll

**File:** `src/components/conversation/message-bubble.tsx` → `CodePre` component (lines ~8965–9031)

**Root cause:** Two distinct problems:

**a) Scrollbar too large**: The inner `<div>` uses `overflow-auto` which falls back to the platform default scrollbar. On macOS/Windows with a wheel mouse or Tauri's webview this renders an oversized native scrollbar.

**Fix:** Add a thin-scrollbar CSS utility. In `src/App.css`, add:
```css
.thin-scrollbar {
  scrollbar-width: thin;       /* Firefox */
  scrollbar-color: var(--border) transparent;
}
.thin-scrollbar::-webkit-scrollbar {
  height: 4px;
  width: 4px;
}
.thin-scrollbar::-webkit-scrollbar-track {
  background: transparent;
}
.thin-scrollbar::-webkit-scrollbar-thumb {
  background-color: var(--border);
  border-radius: 4px;
}
```
Then apply it in `message-bubble.tsx` `CodePre`:
```tsx
// Before
className="overflow-auto max-h-96 ..."
// After
className="overflow-auto max-h-96 thin-scrollbar ..."
```

**b) Background mismatch after horizontal scroll:** Shiki's `@shikijs/rehype` outputs inline background colors on the `<pre>` element when `useBackground: true` (or when the theme sets it implicitly). Even with `useBackground: false`, some themes write a `background-color` directly on the generated `<pre>`. When scrolling right, the `<div>` wrapper ends, but the `<pre>` background continues — this causes a color seam.

**Fix:** In `message-bubble.tsx`, in the `CodePre` inner `<div>`, ensure the background is always set to the card background and override any Shiki-injected backgrounds:
```tsx
// Change the className line for the inner scroll wrapper:
className="overflow-auto max-h-96 thin-scrollbar transition-opacity duration-200 bg-card [&>pre]:!m-0 [&>pre]:!rounded-none [&>pre]:!border-none [&>pre]:p-3 [&>pre]:text-xs [&>pre]:leading-relaxed [&>pre]:!bg-transparent"
```

The key additions are:
- `bg-card` on the scroll container to establish the baseline background
- `[&>pre]:!bg-transparent` to strip any Shiki-injected background from the `<pre>` so the parent's `bg-card` shows uniformly when you scroll right

This is entirely a CSS class change — no logic changes.
