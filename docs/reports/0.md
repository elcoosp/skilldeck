# UX Research Findings: SkillDeck Usability Issues Analysis

## 🎯 Research Overview

### Objectives
**Primary Questions**: 
- Why are users experiencing friction with context injection and skill management features?
- What cognitive barriers prevent users from understanding skill operations?
- How can we reduce error states and improve discoverability?

**Methods Used**: Heuristic evaluation, task flow analysis, cognitive walkthrough

**Findings Coverage**: 9 distinct usability issues affecting core interaction patterns

---

## 📊 Detailed Issue Analysis & Recommendations

---

### Issue 1: Missing Margin Top on Attachments

**Observed Behavior**: When users attach items, the context chips appear flush against the textarea with no visual separation.

**User Impact**: 
- Reduced scanability of attached context items
- Visual crowding increases cognitive load when reviewing message composition
- Users may miss that attachments are present before sending

**UX Analysis**: 
The attachment area serves as a "shopping cart" for context - users need to quickly verify what they're about to send. The current layout violates Fitts's Law by making the interactive chips too close to the text input area, increasing accidental clicks.

**Recommendation**: 
Apply `margin-top: 8px` to the attachments container. This creates visual breathing room and establishes a clear hierarchy between "what I'm typing" and "what I'm attaching".

---

### Issue 2: Clicking File/Skill Below Input Inserts # and @ Symbols

**Observed Behavior**: Toolbar buttons that should trigger pickers instead insert trigger characters directly into the input field.

**User Mental Model Breakdown**:
1. User sees button labeled "Skill" - expects it to open skill picker
2. User clicks button - sees "#" appear in text input
3. User must now either:
   - Recognize this as a trigger and continue typing
   - Delete the character and try a different approach
   - Feel confused about what happened

**Cognitive Friction Analysis**:
This violates the principle of **direct manipulation**. Buttons should perform immediate, predictable actions. Forcing users to understand a command syntax before accessing a feature adds unnecessary learning overhead.

**Recommendation**:
- Buttons should directly open pickers without inserting characters
- The inserted characters should be a fallback for power users who prefer keyboard-driven workflows
- Consider visual distinction: buttons labeled "Insert Skill" vs typing "@" for power users

---

### Issue 3: Arrow Up/Down Navigation Broken in Folder Scope Modal

**Observed Behavior**: Keyboard navigation between "direct children only" and "all nested files" options doesn't work as expected.

**User Expectations**:
- Pressing ↑/↓ should move between the two primary options
- Visual focus indicator should follow keyboard navigation
- Enter should select the focused option

**Accessibility Impact**:
This breaks WCAG 2.1 Success Criterion 2.1.1 (Keyboard). Users who rely on keyboard navigation cannot effectively use this modal.

**Recommendation**:
Implement proper keyboard focus management:
- Trap focus within modal while open
- ↑/↓ cycles between the two option buttons
- Enter selects focused option
- Escape returns to previous view

### Issue 4: # Appears in File Picker Search Input

**Observed Behavior**: Typing "#" inserts the character into the picker's search field, resulting in no matches found.

**User Journey Analysis**:

| Step | User Action | Expected | Actual |
|------|------------|----------|--------|
| 1 | Type "#" to trigger file picker | Picker opens, "#" not in search | Picker opens WITH "#" in search |
| 2 | Begin typing filename | Search filters by filename | Search includes "#" prefix → zero results |
| 3 | User sees "No files found" | Confusion about why search failed | Abandons file attachment or tries again |

**Root Cause**:
The trigger character is being propagated to the picker's internal search state instead of being consumed by the trigger mechanism.

**Recommendation**:
- The "#" trigger should be ephemeral - once the picker opens, the trigger character should be cleared from the input buffer
- Search should start empty, allowing users to type their query fresh

### Issue 5: Sync Registry Button is Janky

**Observed Behavior**: The sync button provides inadequate feedback about its state and operation.

**Heuristic Violations**:
- **Visibility of system status**: No clear indication that sync is in progress
- **Match between system and real world**: "Sync" implies two-way communication, but users may not understand what's being synced
- **Error prevention**: No confirmation before potentially overwriting local state

**Recommendation**:
- Add loading state with spinner and "Syncing..." text
- Show success/failure toast notification with count of updated skills
- Provide last-sync timestamp in tooltip or nearby UI
- Consider adding "Sync in progress" indicator that's non-blocking

---

### Issue 6: "Uninstall" Meaning for Local Skills

**Observed Behavior**: Users are uncertain what "uninstall" does to locally created/modified skills.

**Mental Model Mismatch**:

| User Assumption | Actual Behavior |
|----------------|-----------------|
| "Uninstall removes from my list but keeps files" | ❌ Files are deleted from disk |
| "Uninstall only affects this workspace" | ❌ Personal skills are removed globally |
| "Uninstall just disables, I can re-enable" | ❌ Requires reinstallation from source |

**UX Research Finding**:
Users distinguish between "installed" (from registry) and "created" (local) skills. The same verb creates confusion for two different origins.

**Recommendation**:
- Differentiate terminology: "Remove" for local skills, "Uninstall" for registry-derived skills
- Add confirmation dialog showing exactly which files will be deleted
- Include "Show in Finder/Explorer" option before deletion
- Consider "Move to Trash" pattern with undo capability

---

### Issue 7: Open Folder in Skill Detail Does Nothing

**Observed Behavior**: Button implies ability to view skill files, but no action occurs when clicked.

**User Expectation**: 
"I want to see where this skill lives on my computer to edit it directly."

**Gap Analysis**:
The button exists but lacks implementation - this creates a **false affordance**. Users learn to distrust UI elements when expectations aren't met.

**Recommendation**:
- Either implement the functionality (open system file explorer at skill location) OR
- Remove the button entirely
- If keeping for future release, add disabled state with tooltip: "Coming soon - edit skills directly"

---

### Issue 8: Skill Tab Cards Inconsistent Layout

**Observed Behavior**: Cards have varying heights and information density, causing horizontal overflow and scrollbars.

**Visual Design Principles Violated**:
- **Consistency**: Similar content should appear in similar containers
- **Hierarchy**: Information priority should be visually communicated
- **Whitespace**: Inconsistent spacing creates visual noise

**User Impact**:
- Reduced scanability when browsing skills
- Horizontal scrollbar suggests content is wider than viewport - users may miss information
- Cognitive load increases as users must decode each card's layout individually

**Recommendation**:
- Standardize card dimensions and information hierarchy
- Truncate descriptions with ellipsis after 2-3 lines
- Use consistent metadata placement (author, tags, scores)
- Implement responsive grid that adapts to panel width
- Consider list view alternative for power users who prefer density

---

### Issue 9: "Platform Not Configured" Error with No Fix Guidance

**Observed Behavior**: Users clicking sync button receive error with no explanation of how to resolve.

**Error Message Analysis**:

| Component | Current | Recommended |
|-----------|---------|-------------|
| What happened | "Platform not configured" | "Can't connect to skill registry" |
| Why it happened | ❌ Missing | "You need to connect to SkillDeck Platform first" |
| How to fix | ❌ Missing | "Open Settings → Platform → Enable sync" |
| Visual treatment | Generic error | Actionable callout with link |

**UX Best Practices for Errors**:
1. **Acknowledge** what went wrong
2. **Explain** why it happened (in user terms)
3. **Provide** clear next steps
4. **Offer** direct action when possible

**Recommendation**:
```typescript
// Error UI pattern
<Alert>
  <AlertTitle>Platform sync unavailable</AlertTitle>
  <AlertDescription>
    You need to configure platform settings before syncing skills.
    <Button variant="link" onClick={() => openSettings('platform')}>
      Open platform settings
    </Button>
  </AlertDescription>
</Alert>
```
