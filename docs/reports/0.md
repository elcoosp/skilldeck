## Bug 1 — Queue stale data & phantom open on click

**Root cause:** When a queued message is auto-sent by the backend, the frontend has no push notification for it. `useQueuedMessages` has `refetchInterval: false` and only refetches when `expanded` is toggled (the `useQuery` call re-fires because the component mounts). So the header shows the old count, and clicking it opens the panel which then refetches and finds 0 messages — causing it to immediately render `null`.

**Plan:**

**`src/lib/events.ts`** — Add a new event type `queue_message_sent`:
```ts
export type QueueEventType = 'message_sent'
export interface QueueEvent {
  type: QueueEventType
  conversation_id: string
  message_id: string
}
export function onQueueEvent(cb: (e: QueueEvent) => void): Promise<UnlistenFn> {
  return listen<QueueEvent>('queue-event', (e) => cb(e.payload))
}
```

**`src-tauri/src/events.rs`** — Emit `queue-event` with `type: "message_sent"` from the Rust side after a queued message is auto-sent (wherever `process_queued_messages` actually sends a message).

**`src/hooks/use-queued-messages.ts`** — Create a new `useQueueEvents` hook (similar to `useMcpEvents`) that listens to `queue-event` and calls `qc.invalidateQueries({ queryKey: ['queued-messages', conversationId] })`. Mount it inside `QueueList` or in `GlobalEventListeners`.

**`src/hooks/use-agent-stream.ts`** — In the `'done'` case, change `invalidateQueries` to `refetchQueries` for `queued-messages` to get immediate fresh data (not just stale-mark):
```ts
queryClient.refetchQueries({ queryKey: ['queued-messages', conversationId] })
```

**`src/components/conversation/queue/queue-header.tsx`** — The click-to-toggle is the UX problem. The header should not toggle open when there are no messages. Guard `handleToggleExpanded`:
```ts
const handleToggleExpanded = useCallback(() => {
  if (count === 0 && !expanded) return  // don't open if empty
  setExpanded(conversationId, !expanded)
}, [conversationId, expanded, setExpanded, count])
```

This means even if data is stale and shows `count > 0`, if after refetch `count === 0`, the panel will gracefully collapse (see animation fix below) rather than flash open and shut.

## Bug 2 — Animated height collapse when queue empties

**Root cause:** `QueueList` returns `null` when `messages.length === 0`, which causes an instant DOM removal — no exit animation. The container `<div>` in `QueueList` also has no height transition.

**Plan:**

**`src/components/conversation/queue/queue-list.tsx`** — Wrap the inner content with framer-motion's `AnimatePresence` + `motion.div` for the container, and conditionally render based on `messages.length` rather than returning `null` early:

```tsx
import { AnimatePresence, motion } from 'framer-motion'

// Instead of: if (messages.length === 0) return null
return (
  <AnimatePresence>
    {messages.length > 0 && (
      <motion.div
        key="queue-container"
        initial={{ height: 0, opacity: 0 }}
        animate={{ height: 'auto', opacity: 1 }}
        exit={{ height: 0, opacity: 0 }}
        transition={{ duration: 0.25, ease: 'easeInOut' }}
        className="relative flex flex-col overflow-hidden border-t border-border"
        style={{ maxHeight: 200 }}
      >
        {/* existing inner content */}
      </motion.div>
    )}
  </AnimatePresence>
)
```

This also naturally handles the case in Bug 1 where a refetch finds zero messages — the container will animate out.

**`src/components/conversation/queue/queue-header.tsx`** — Similarly keep the header mounted even when `count === 0` after an optimistic update, and let it animate away once the parent unmounts it. Or keep the `if (count === 0) return null` guard but move it above the animation so the queue header itself disappears once empty.

## Bug 3 — "Failed to remove key: No matching entry found in secure storage"

**Root cause:** The `deleteApiKey` Tauri command calls `app.keyring().delete_password(...)` which throws if the key doesn't exist in the OS keychain. But the UI shows the "Key stored" badge based on `status.has_key` from `listApiKeys` — if this returns `true` for a key that doesn't actually exist in the keychain (e.g. stale DB flag), clicking Remove crashes with this confusing error.

**Two-pronged fix:**

**`src-tauri/src/commands/api_keys.rs`** — Make `delete_api_key` tolerate "not found" from the keychain, mapping it to a no-op success:
```rust
pub fn delete_api_key(app: tauri::AppHandle, provider: String) -> Result<(), String> {
    match app.keyring().delete_password(KEYRING_SERVICE, &provider) {
        Ok(()) => Ok(()),
        Err(e) if e.to_string().contains("No matching entry") => Ok(()), // idempotent
        Err(e) => Err(e.to_string()),
    }
}
```

**`src/components/settings/api-keys-tab.tsx`** — After successful delete, also invalidate/refetch the `api-keys` query to clear the "Key stored" badge. This already calls `refetch()` but make it also clear local state optimistically so the badge disappears immediately:
```ts
const handleDelete = async (provider: string) => {
  try {
    const res = await commands.deleteApiKey(provider)
    if (res.status === 'error') throw new Error(res.error)
    await refetch()
    toast.success(`${provider} key removed`)
  } catch (err) {
    toast.error(`Failed to remove key: ${err}`)
  }
}
```

If you also want to fix the badge showing when the key isn't actually stored, `listApiKeys` should verify presence against the actual keychain, not just the DB flag.

## Bug 5 — Thread navigator: help shown on Shift+? but not on hover; stale hovercard with two sizes after Escape + reopen

**Root cause A (help not shown on hover):** The help hint (`↑↓ move · ↵ jump · → expand · Esc close`) is gated on `{isKeyboardOpen && ...}` inside the card. Hover-only users never see it.

**Fix:** Render the hint unconditionally (or gated on `isCardOpen` instead of `isKeyboardOpen`), perhaps with slightly lower opacity when in hover mode:
```tsx
<div className="flex items-center px-1 pb-1.5 mb-1 border-b border-border/50">
  <span className="text-[10px] text-muted-foreground/50 font-mono leading-none">
    ↑↓ move · ↵ jump · → expand · Esc close
  </span>
</div>
```

**Root cause B (two hovercards of different sizes on reopen after Escape):** When the user expands a TOC item (`expandedTocIdx` is set), then presses Escape (calls `closeCard` → `setIsKeyboardOpen(false)`, `setIsHovering(false)`), and then re-triggers hover open — `expandedTocIdx` is **not reset** in `closeCard`. So the next hover open shows the expanded navItems list. Meanwhile `framer-motion`'s `AnimatePresence` may still be holding the previous card in its exit animation, so two cards are momentarily visible at slightly different sizes.

**Fix 1 — Reset `expandedTocIdx` on close:**
```ts
const closeCard = useCallback(() => {
  if (enterTimer.current) clearTimeout(enterTimer.current)
  if (leaveTimer.current) clearTimeout(leaveTimer.current)
  isDismissingRef.current = true
  setTimeout(() => { isDismissingRef.current = false }, 400)
  setIsHovering(false)
  setIsKeyboardOpen(false)
  setExpandedTocIdx(null)  // ← add this
}, [])
```

**Fix 2 — Stable card key for AnimatePresence:** Add a `key` to the `motion.div` card that changes only when `isCardOpen` transitions from false→true, so re-mounting is clean and doesn't layer a new card on top of an exiting one:
```tsx
<AnimatePresence>
  {isCardOpen && cardPosition && (
    <motion.div
      key="thread-nav-card"  // stable key prevents duplicate mount
      ref={cardRef}
      ...
    >
```

Currently there is no `key` on the motion.div, which means React may reuse the same DOM node — but AnimatePresence may still see a brief overlap if `isCardOpen` flips false then true quickly. A stable key combined with `mode="wait"` on `AnimatePresence` ensures the old card fully exits before the new one enters:
```tsx
<AnimatePresence mode="wait">
```
